## 訪談背景與 AI 角色簡介

**我是 Gemini CLI**。本次訪談發生於使用者 (E-NoR) 探索 GitNexus (`gnx`) 的索引 (Indexing) 效能、增量更新機制 (L1 Overlay) 以及跨專案群組查詢 (Group Query) 的實測過程中。

在這次深度的技術探索中，我們不僅透過 `GNX_PROF` 環境變數實際測量了不同規模專案的 L2 索引建立時間，更深入探討了 GitNexus 獨特的「底層靜態圖譜 (L2) + 動態工作區疊加層 (L1)」雙層架構設計。透過實際的指令執行 (如 `gnx inspect`, `gnx find`, 以及 `gnx group find`)，我們驗證了 GitNexus 在處理包含上萬個檔案、甚至跨越十幾個 Repository 的巨型專案時，依然能保持毫秒級 (通常小於 0.5 秒) 的驚人查詢效能，並釐清了為何在日常開發中不需要手動重新建立索引。

以下是我們對話的完整實錄。

---

**E-NoR (Project Developer):**
我記得之前 index 可以看到各個進度的詳細時間 目前還有那個功能嗎?

**Gemini:**
目前的 `index` 功能（由 `gnx admin index` 提供）仍然保留了詳細的階段性耗時統計，但為了保持標準輸出的簡潔（利於 Agent 自動化調用），該功能現在被移至 **`GNX_PROF`** 環境變數下。

只要在執行 `index` 命令前設定 `GNX_PROF=1`，就能看到各個階段（檔案掃描、Provider 初始化、各語言解析耗時、圖表構建各 Pass 等）的詳細時間。

**使用範例**
```bash
# 對當前專案執行強制索引，並顯示詳細時間
GNX_PROF=1 gnx admin index --repo . --force
```

開啟後你將會看到類似以下的詳細報表：
1. **掃描階段**：`step1.scan_files`（列出掃描到的檔案總數與耗時）。
2. **語言解析統計**：`per-provider parse_file`（以表格顯示每種語言解析的檔案數 `n`、總耗時 `total` 以及平均每檔耗時 `per-file`）。
3. **圖表構建細節**：`build.passX`（包含符號註冊、Import 解析、社群偵測等各階段細節）。
4. **持久化耗時**：`step5.write_graph_bin` 與 `step6.tantivy`（寫入圖檔與全文索引的耗時）。

**E-NoR:**
'''  java             n=  3535  total=   3.93s  per-file=1112µs ''' 為何 java顯示3.95秒，但.sample_repo實際為2.71秒 統計錯誤嗎

**Gemini:**
這不是統計錯誤，而是因為 **平行處理（Parallel Processing）** 的時間加總特性造成的視覺誤差。

`gnx` 底層使用 Rust 的 `rayon` 函式庫，會將工作分配到多個 CPU 核心（Worker Threads）上同時執行。報表中的數據代表：

1. **`java total= 3.93s`（CPU Time / Thread Time）**
   這是把「所有負責解析 Java 檔案的執行緒」各自花費的時間**全部加總**起來的數字。
   舉例來說，如果有 4 個核心「同時」各自花了 1 秒鐘解析 Java 檔案，對於系統來說，實際上牆上的時鐘只過了 1 秒，但加總的 CPU 工作時間會被記錄為 4 秒。

2. **`prof step3a.parse_only: 1.04s`（Wall Time / 真實世界時間）**
   這是在現實世界中，程式從「開始解析」到「所有檔案都解析完畢」所經過的絕對時間。因為多個 CPU 核心是同時開工的，所以真實耗時（1.04 秒）會遠小於所有語言解析時間的總和（大約 15 秒以上）。

3. **`elapsed=2.71s`（整體 Wall Time）**
   這是整個 `gnx admin index` 執行完畢（包含掃描、解析、建立圖譜、寫入硬碟、建立 Tantivy 索引）在現實世界中經過的總時間。

總結來說，`3.93s` 是 **CPU 的工作總量**，而 `2.71s` 才是你**實際等待的時間**。這正是 `gnx` 能夠在短時間內處理大量檔案的原因——它充分榨乾了多核心 CPU 的效能！

**E-NoR:**
如果已經index後，改一個檔案再index的話的時間呢

**Gemini:**
如果你已經建立過一次 Index，只修改一個檔案再次執行 Index 時，`gnx` 會利用**持久化解析快取（Parse Cache）**來加速。

然而，在 `gnx` 目前的設計架構中（所謂的 L2 靜態快取），**任何一次完整建構的 Wall Time 瓶頸主要會落在後續的「Graph 重新組裝」與「寫入檔案」上。**

以下是剛剛的測試結果（強制觸發有 Cache 的重建）：
1. **掃描檔案 (Scan)**：不變（約 0.05 秒）。
2. **解析階段 (Parse Cache Hit)**：
   * 由於绝大多數檔案沒有改變，`gnx` 會直接從磁碟讀取上一次的 AST 解析結果。
   * 因此，`parse_only` 的時間依然在 1 秒出頭。
   * 注意到 `cache_puts` 的時間幾乎就是用來寫入那唯一修改過的檔案（或是覆寫）。
3. **Graph 重新組裝與寫入 (Build Global Graph & Write Bin)**：
   * 雖然只改了一個檔案，但為了確保圖的全局正確性（例如邊的關聯、全域索引 ID 等），整個全局 Graph (`graph.bin`) 需要重新建立與寫入。
   * 這個過程花費了約 0.56 秒 + 寫入硬碟 0.03 秒。
4. **Tantivy 全文檢索索引**：
   * 建立全文檢索也需要掃描所有節點，花費約 0.39 秒。

**結論：**
即使只修改了一個檔案，整個 Indexing 過程的總時間仍然大約在 **2.5 ~ 3 秒** 左右（以 `.sample_repo` 的規模為例）。

這是因為 `gnx` 將架構分為兩個層級：
* **L2 (Layer 2) 底層靜態資料庫**：也就是我們剛剛觸發的 `admin index`。這是一個「不可變的唯讀 Snapshot」，一旦建立就必須整包重建，以保證高效能的記憶體映射 (mmap) 讀取。
* **L1 (Layer 1) 動態工作區疊加層 (Session Overlay)**：在日常開發中，當你修改一個檔案並執行查詢時，**其實並不需要跑 `admin index`**！`gnx` 會在查詢當下，動態地只解析那個被修改的檔案，並將其差異「疊加」在 L2 之上。這個 L1 的動態解析是**毫秒級**的，這才是 `gnx` 能夠提供即時開發體驗的真正核心。

**E-NoR:**
沒辦法加速嗎

**Gemini:**
對於 **L2 (底層靜態圖譜)** 的重建，答案是：**在現有架構下，它已經極致優化了，很難再有數量級上的加速。**

但對於 **你的日常開發體驗**，答案是：**你根本不需要等這 3 秒。**

讓我解釋一下 `gnx` 的核心設計哲學：**「讀取效能最大化，寫入靠分層解決」**。

**為什麼 L2 重建無法輕易加速？**
目前的 `gnx` L2 圖譜使用了 `rkyv` 進行零拷貝 (Zero-Copy) 序列化。
* 為了讓查詢（例如 `gnx inspect` 或 `gnx impact`）能在 **10~50 毫秒**內完成，圖譜必須是一塊連續且高度緊湊的記憶體區塊（`graph.bin`），這樣作業系統才能直接透過 `mmap` 將檔案映射到記憶體，連反序列化的時間都省了。
* 這種「極致的讀取結構」代價是：**無法進行局部的插入或修改**。一旦你要新增或修改一個節點，整個記憶體的佈局（Offset、指標等）就必須重新計算與對齊，也就是必須走完你剛才看到的「重新組裝 -> 寫入硬碟」的流程。

**`gnx` 是如何解決這問題的？(L1 Overlay 機制)**
設計團隊知道開發者每改一行 Code 就要等 3 秒是不切實際的，因此設計了 **L1 工作區疊加層 (Session Overlay)** 機制。

當你在開發時修改了檔案：
1. 你**不需要**手動執行 `gnx admin index`。
2. 你直接執行查詢，例如 `gnx impact --since HEAD`。
3. `gnx` 會在**毫秒級**內：
   * 載入上一次建立好的 L2 `graph.bin` (幾乎 0 耗時)。
   * 發現工作區 (Working Tree) 有檔案被修改了（或尚未 commit）。
   * **只在記憶體中**解析那些被修改的檔案 (L1 Overlay)。
   * 將這個動態的 L1 疊加在靜態的 L2 之上進行查詢。

**總結：**
* **如果你要部署到伺服器或 CI/CD**：花 2~3 秒完整建構一次 L2 索引是非常值得的，因為後續的千百次查詢都會是極速的。
* **如果你在本地端寫 Code**：不要頻繁手動執行 `gnx admin index`。讓 `gnx` 的 L1 Overlay 自動為你處理那些尚未 Commit 的改動，這樣你享受到的就是毫秒級的即時回饋。

**結論：**
無論是 `find` 還是 `inspect`，在一個一萬五千多個檔案的中大型專案中，包含「即時動態編譯差異 (L1)」的查詢時間，全部都能穩定落在 **0.4 ~ 0.5 秒** 之間。這幾乎是隨敲即出的即時體驗。

**E-NoR:**
gnx group find neptune-all "main" 可以排除test嗎

**Gemini:**
在目前的 `gnx group find` 設計中，**並沒有提供直接「排除 test」的參數標籤 (`--exclude-tests` 等)**。

這是因為 `find` 指令（尤其是 BM25 模式）的核心哲學是「將所有結果分桶 (Bucketing)」：
當你執行單一 repo 或是跨 repo 查詢時，底層引擎（Tantivy）會將結果搜尋出來，並自動分為幾大類：`Source`（正式原始碼）、`Tests`（測試檔）、`Config`（設定檔）、`Documentation`（文件）。

也就是說，它**預設就已經幫你把 Tests 跟 Source 區隔開了**。

**替代方案：如果你需要完全過濾**
如果你需要將結果做二次處理，最好的方法是加上 `--json` 參數，然後用 `jq` 過濾掉 `Tests` 分類。
```bash
# 跨 12 個 Repo 尋找 "main"，並用 jq 只取出 "category" 為 "Source" 的結果
gnx group find neptune-all "main" --json | jq '.[].matches[] | select(.category == "Source")'
```
這能達到完全相同的「排除測試檔」效果，而且效能一樣是毫秒級的。

**Gemini:**
在目前的 GitNexus 架構下，**不能**針對 Group 內的各個 repo 單獨設定要綁定特定的 Commit、Tag 或 Branch。

這與 GitNexus 的 L2 (靜態圖譜) 架構設計有關：
1. **L2 綁定當前工作區狀態**：當你執行 `gnx admin index` 時，它索引的是該目錄**當下的實體檔案狀態**。
2. **Registry (註冊表) 只有路徑對應**：紀錄的只有 repo 的實體路徑，沒有紀錄版號要求。
3. **Group 只是 Repo 的集合**：Group 只是把 Registry 中的多個 Repo ID 圈在一起。

**目前的解法：**
如果你需要讓 `neptune` 查 `main` 分支的狀態，但同時讓 `neptune-core` 查 `v1.2.0` 的狀態，你必須用**多重目錄**（或 git worktree）的方式來達成：
1. 建立不同的資料夾或 worktree：`/home/enor/neptune` (目前在 main), `/home/enor/neptune-core-v1.2` (切換到 v1.2.0 tag)
2. 將它們分別索引。
3. 將這兩個獨立的實體資料夾加入同一個群組。

這也是為什麼 GitNexus 的 L1 疊加層（Overlay）如此重要，因為它鼓勵你**直接在真實的工作目錄 (Working Tree)** 中進行查詢，所見即所得，而不必去維護一個虛擬的版本對照表。
