**E-NoR:**
在你的認知了解上，gnx 有可能可以做到'完整證明 Rayon closure 裡沒有共享 mutable state' 嗎?

**Codex:**
在我的理解裡，**gnx 可以輔助接近這件事，但不太適合單獨做到「完整證明」**。

原因是「Rayon closure 裡沒有共享 mutable state」本質上接近 Rust 型別系統 / borrow checker / trait bound 的靜態保證問題。完整證明通常要依賴：

- `Send` / `Sync` trait bound
- closure capture 分析
- interior mutability：`Mutex`、`RwLock`、`Cell`、`RefCell`、`Atomic*`
- `unsafe`
- 全部 call graph 內是否有 hidden mutable global / thread-local / FFI side effect

`gnx` 如果只是一個 code graph / symbol graph，它可以做到：

- 找出 closure 內呼叫了哪些函式
- 找出 `pipeline.parse_file_raw` 的實作與 callers
- 找出相關型別如 `AnalyzerPipeline`、`LanguageProvider`
- 找出 `unsafe`、`RefCell`、`Mutex`、`static`、`thread_local!` 等風險點
- 幫 review 者縮小需要人工驗證的區域

但「完整證明」要更強：它需要理解 Rust type checking 結果、MIR/borrow facts，甚至整合 `rustc` 的 trait resolution。單靠目前這種 graph-level analysis，通常不夠。

比較務實的答案是：

- **可以做到高信心輔助判斷**：例如 gnx 找出 closure capture + call graph + mutable/global/unsafe 風險點。
- **不能單獨取代 rustc**：如果程式能通過編譯，Rust 已經證明了資料競爭層面的 `Send`/`Sync` 約束；gnx 更像是把「為什麼可信」整理給 reviewer。
- **若要完整證明，gnx 需要接 rustc metadata/MIR**，而不是只靠 tree-sitter / symbol graph。

具體來說，`gnx review` 幫我省的是：

- 不必先完整掃整個 PR 以外的模組
- 很快確認主要風險集中在 `impact_with_baseline`
- 知道 blast radius 只有少量 caller，沒有先冒出高風險訊號

但這個 PR 只改一個檔案、99 行，diff 本身很小，所以 `gnx` 的效益不是最大。真正省很多 token/時間的情境會是：

- PR 改很多檔案
- call graph 跨 crate / 跨模組
- 有多個入口點或 CLI command path
- 需要快速判斷哪些 diff 是機械變更、哪些是行為變更
- 要找既有 helper / duplicate logic

這次我的結論是：  
`gnx` 有效降低定位成本，但沒有大幅取代人工 review。對小 PR 是加速器；對大 PR 或跨模組變更，價值會更明顯。

**Codex:**
有重跑，同一個指令結果如下：

```bash
/usr/bin/time -f 'wall=%e' gnx review --repo . --since origin/main --format toon
```

三次結果：

| Run | `elapsed_ms` | wall-clock | 備註 |
|---|---:|---:|---|
| 1 | 918ms | 2.20s | `l1.refreshed written=188` |
| 2 | 899ms | 2.18s | `l1.refreshed written=188` |
| 3 | 900ms | 2.21s | `l1.refreshed written=188` |

跟前一次 review 時的：

```text
elapsed_ms: 3193
```

相比，現在 warm-ish run 的內部時間約 **0.9s**，少了很多。

判斷：前一次 **3.19s 很可能含有 cold-ish 成本**，例如首次 L1 refresh / graph 準備 / OS cache 未熱。  
但現在每次還是固定看到：

```text
l1.refreshed written=188 failed=0
```

所以目前 wall-clock 約 2.2s 裡面，還有一段不在 `elapsed_ms` 內或不完全被 review summary 計入的前置 refresh/startup 成本。真正 review aggregator 本體看起來約 **900ms**。

---

**E-NoR:**
這是因為sandbox緣故嗎?假設你跑在sandbox外呢?

**Codex:**
這次「第一次失敗」是 sandbox 緣故：

```text
Error preparing index for .: L1 overlay refresh: Read-only file system
```

`gnx review` 會刷新 L1 overlay，需要寫到 gnx 的索引/overlay 目錄。sandbox 內那個位置不可寫，所以失敗。後來我用 escalated/outside sandbox 跑，就成功了。

但成功後的時間：

```text
l1.refreshed written=188 failed=0
elapsed_ms: ~900
wall: ~2.2s
```

這些已經是在 sandbox 外跑的結果。也就是說：

- **Read-only failure**：是 sandbox 造成。
- **每次 2.2s wall-clock**：不是單純 sandbox 造成，因為已經 outside sandbox。
- **`elapsed_ms` 約 900ms**：比較像 `gnx review` aggregator 本體時間。
- **wall 比 elapsed 多約 1.3s**：多半是前置成本，例如 CLI startup、graph/registry/index resolve、L1 overlay refresh、git baseline 檢查等。

另外每次都顯示：

```text
l1.refreshed written=188
```

這代表它不是完全 no-op warm run；它每次還在重寫 188 個 L1 overlay 項目。這應該是目前 2.2s wall-clock 的主要可疑來源之一。

> **後續修正註記（2026-05-19）：** 這段描述的是訪談當時的行為。相關設計後續已修正：PR #161 將 L1 overlay 的 manifest/version 寫入合併成批次，並平行化 fragment 處理，降低每個 dirty file 重複寫入的成本；PR #167 加入 HEAD-SHA sidecar，讓乾淨 worktree 且 HEAD 與索引一致時可以跳過全源碼 mtime walk。簡言之，修復方式是避免乾淨讀取時不必要的 L1 churn，並把仍需處理的 dirty-file refresh 合併批次化。
