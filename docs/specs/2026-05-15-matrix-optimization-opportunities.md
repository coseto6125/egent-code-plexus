## A. Quality & Depth Issues on ✓ Cells

### A1. Exports — Go, Ruby, C, Dart shallow visibility

| Lang | Issue | Evidence | Optimization |
|------|-------|----------|---|
| **Go** | Exports detected by leading-uppercase only; struct field visibility not propagated | `go/parser.rs:~90` `is_exported = name.chars().next().is_some_and(\|c\| c.is_uppercase())` | Add per-struct-field visibility capture in `queries.scm`; emit `is_exported` on each `Property` node |
| **Ruby** | `public`/`private`/`protected` keywords parsed; `attr_reader`/`attr_writer`/`attr_accessor` metaprogramming not captured | `ruby/parser.rs` & `queries.scm`: no attr_* patterns | Add tree-sitter query matching `call_expression` whose receiver-name starts with `attr_`; emit derived visibility on the attribute Property |
| **C** | Exports inferred only from header inclusion; `extern` / file-scope `static` / `__attribute__((visibility(...)))` not parsed | `c/parser.rs`: no extern handling | Tree-sitter-c queries for `extern_specifier`, file-scope `static`, GCC visibility attributes |
| **Dart** | Only `library` directive captured; per-method/per-field underscore convention (`_name` = private) not flagged | `dart/parser.rs`: library-level only | Add per-symbol underscore check at parse-time; set `is_exported: false` when name starts with `_` |

**Est. LOC**: ~60-90 total. High-value: Go struct-field exports + Dart underscore convention are standard idioms.

### A2. Imports — TS/JS re-export alias tracking

| Lang | Issue | Evidence | Opportunity |
|------|-------|----------|---|
| **TypeScript** | `export { X as Y } from 'y'` parsed as export-only; alias not preserved on emitted RawImport | `typescript/queries.scm:111-130` captures named-import alias but re-export form drops it | Extend queries to capture re-export aliases; consider new `RelType::ReExports` or wire alias into existing RawImport |

### A4. Heritage — Ruby mixins missing

| Lang | Issue | Evidence | Notes |
|------|-------|----------|---|
| **Python** | Multiple inheritance `class C(A, B)` lumps all bases into `heritage: vec!["A","B"]` without ordering | `python/parser.rs:~280` `heritage.extend(...)` | Sufficient for MRO reconstruction downstream — skip |
| **Ruby** | Only single parent (`class C < Parent`) captured; `include Module` / `extend Module` mixins NOT tracked | `ruby/queries.scm` & `parser.rs`: no mixin patterns | Add tree-sitter queries for `include_statement` and `extend_statement`; emit as secondary heritage edges |

**Est. LOC**: ~80 (Ruby mixins only). Medium value for Ruby metaprogramming inference.

## C. Cross-Cutting Infrastructure Opportunities

### C1. Shared `is_exported` utility

Every language reimplements its own export-detection logic:
- Go: uppercase first-letter check (`go/parser.rs:~90`)
- Java: `public`/`protected` keyword (`java/queries.scm:3-6`)
- Python: underscore-prefix inversion (`python/parser.rs:~167`)
- Rust: `pub` keyword check (parser-side)
- Solidity: visibility keywords (state vars + funcs)

**Optimization**: Add `framework_helpers::visibility::{by_keyword, by_convention}` utilities. Single source of truth for export conventions, testable in isolation.

**Est. LOC**: ~80. Benefit: DRY; centralizes a recurring policy.

---

### C2. Framework detector registry refactor

**Rationale**: `framework_confidence.rs` is flat constants; per-language registration is scattered across parser files.

**Current state**:
- Python FastAPI/Django/Celery in `python/parser.rs:380+`
- Rust Axum/Actix in `rust/parser.rs:200+`
- TypeScript Express/NestJS in `typescript/parser.rs:250+`
- Java Spring in `java/parser.rs` + `frameworks.scm`

**Optimization**: Introduce `FrameworkDetector` trait in `framework_helpers.rs`:
```rust
pub trait FrameworkDetector {
    fn detect(&self, file: &LocalGraph) -> Vec<RawFrameworkRef>;
}
```
Concrete impls: `PyFastAPIDetector`, `RustAxumDetector`, ... Registered in one module so Wave 2 (B1-B4) adds detectors without editing per-language parsers.

**Est. LOC**: ~150 refactor + ~20 per new detector. **High value — unblocks parallel B-task dispatch.**

---

### C3. Receiver-type binding generalization (post-Wave 1)

After Wave 1 extends Constructor Inference to 13 langs, extract reusable patterns from `python/receiver_types.rs`:
- Scope stack (var → type) generic over language
- Enclosing function/class lookup (shared with `framework_helpers.rs`)
- Type-annotation AST patterns (per-lang submodule)

Generalize into `crates/graph-nexus-analyzer/src/receiver_binding/{mod,scope,python,java,typescript}.rs`.

**Est. LOC**: ~200 refactor (post-Wave 1). Deferred to Wave 3.

---

### C4. NodeKind extension: framework-semantic kinds

**Rationale**: Current 14 NodeKind variants don't expose framework semantics. LLM asking "what are the endpoints?" must reconstruct from Route + handler names.

**Optimization**: Add `framework_kind: Option<FrameworkKind>` to RawNode:
```rust
pub enum FrameworkKind {
    SpringBean, FastAPIEndpoint, ExpressRoute,
    AxumHandler, /* ... */
}
```
Populate via `framework_helpers.rs` when emitting RawFrameworkRef — attach to matching Function/Method nodes.

**Est. LOC**: ~100. Clarifies framework-semantic nodes without breaking existing schema.

---

### C5. Tree-sitter query deduplication

Audit showed import patterns mostly DON'T overlap across languages (each grammar has unique node types). Minor overlap exists within TS/JS pair but not significant. **Skip** unless future survey shows >20% duplication.

## E. Known Non-Opportunities (Out of Scope)

- **N1**. Type inference for untyped langs (Ruby, Lua, Bash) — runtime-dependent.
- **N2**. Multi-level / transitive imports — Pass 2 resolver already handles.
- **N3**. Macro/template expansion (C++, Rust) — AST-based; can't expand macros.
- **N4**. Config-file parsing for 17 langs — most (Docker, YAML, HCL) are config-only.
- **N5**. Cross-lang polymorphism (trait objects, interfaces) — type-system specific.
