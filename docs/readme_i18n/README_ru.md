Автономные агенты по написанию кода выполняют **20–50 структурных запросов на задачу**. Все эти запросы адресованы инструментам, созданным для людей: боковые панели IDE, демоны, требующие прогрева, вывод, отформатированный для чтения. Это несоответствие проявляется в трёх конкретных режимах сбоя:

1. **Расход токенов** — дамп `grep` возвращает 400 строк там, где агенту нужны 10 символов
2. **Сломанные рефакторинги** — пропущенный вызывающий код остаётся незамеченным, потому что резолвер угадал неверно
3. **Придуманные зависимости** — когда статический анализ не может добраться до ребра, инструмент его изобретает

`ecp` создан для устранения всех трёх проблем.

| Режим сбоя | Ответ `ecp` |
|---|---|
| Контекстное окно переполнено сырым выводом поиска | **TOON / компактный JSON** — только символы, строки и рёбра; без балласта |
| Пропущенный вызывающий код, скрытая поломка | **`impact`** — точный радиус взрыва по реальным рёбрам вызовов и наследования |
| Придуманная зависимость в рассуждениях агента | **Записи `BlindSpot`** — типизированные честные пробелы, вокруг которых агент может проложить маршрут |
| Граф слепнет за пределами основного языка | **31 язык** — сервисный код, IaC, SQL, смарт-контракты в одном обходе |

## 🎯 Принципы проектирования

Каждое проектное решение имеет единственный источник: *что на самом деле нужно принимающему агенту?*

**Вывод — это структура данных.** TOON и компактный JSON содержат только то, что агенту нужно для следующего решения. Никаких прозаических резюме. Никакого визуального оформления. Никаких заголовков разделов, пожирающих бюджет контекста. Форматы по умолчанию уже являются правильным выбором для большинства LLM-подсказок.

**Без состояния. Без прогрева.** Каждый вызов выполняет `mmap` файла графа `rkyv` с нулевым копированием и завершается. **~140–170 ms на запрос, включая запуск процесса.** Никакого демона для поддержания в рабочем состоянии. Никакой фазы прогрева. Никакого пути восстановления «сервер упал, перезапустите». Агент может выполнять 50 запросов на задачу, не платя за загрузку процесса.

**BlindSpot вместо галлюцинаций.** Когда `ecp` не может статически разрешить место вызова — динамическая диспетчеризация, рефлексия, неразрешённый импорт — он создаёт запись `BlindSpot`: именованный, типизированный, явный пробел в графе. Агенты могут обходить известные неизвестности. Они не могут оправиться от уверенной фабрикации.

**Полиглотность по умолчанию.** 31 язык на структурной глубине. Сервисный код, Dockerfiles, GitHub Actions, Terraform, SQL, Move, Solidity — один обход охватывает все уровни. Отсутствие переключения языков означает отсутствие слепых пятен графа.

### Холодная индексация в 60× быстрее, чем у GitNexus

Измерено на кодовой базе [gitnexus](https://github.com/abhigyanpatwari/GitNexus) TypeScript · `scripts/parity/benchmark_vs_gitnexus.py`:

| Фаза | ecp (Rust) | gitnexus (Node) | Ускорение |
|---|---|---|---|
| **Холодная индексация** | **~970 ms** | ~58 s | **60×** |
| **Контекст символа** | **~70 ms** | ~430 ms | **6×** |
| **Радиус взрыва** | **~70 ms** | ~460 ms | **6×** |
| **Cypher-запрос** | **~70 ms** | ~400 ms | **5×** |

*Задержка `ecp` включает полный запуск процесса (без демона). GitNexus (v1.6.5) измеряется на прогретом проиндексированном репозитории.*

### Масштаб: `.sample_repo` — 22,645 файлов, 25 языков, полиглотный корпус 2.1 ГБ

**Ввод данных:**

| Метрика | Значение |
|---|---|
| Проиндексировано файлов | **22,645** на 25 обнаруженных языках |
| Холодный ввод | **2.60 s** (разбор + разрешение + сериализация) |
| Инкрементальный ввод | **4.9 ms** (обход хешей xxh3_64, ноль изменённых файлов) |

**Задержка на запрос, включая запуск процесса:**

|---|---|---|
| `summary` | **1.4 ms** | mmap реестра — минимальное чтение |

| `summary --detailed` | **143.4 ms** | полный реестр + оценка уверенности по фреймворкам |
| `impact --direction down` | **145.0 ms** | BFS по рёбрам Calls / Extends |

## 📦 Установка

Готовые бинарные файлы поставляются с каждым GitHub Release. Скрипты установщика прибегают к сборке из исходников через cargo только при отсутствии подходящего ассета.

```bash
# Linux / macOS
curl -sSfL https://github.com/coseto6125/egent-code-plexus/releases/latest/download/install.sh | sh

# Windows PowerShell
iwr https://github.com/coseto6125/egent-code-plexus/releases/latest/download/install.ps1 -UseBasicParsing | iex

# Прямо через cargo (без обёртки установщика)
cargo install --git https://github.com/coseto6125/egent-code-plexus egent-code-plexus --bin ecp --locked
```

Сборка из исходников с оптимизацией под CPU:

```bash
repo=https://github.com/coseto6125/egent-code-plexus
RUSTFLAGS="-C target-cpu=native" cargo install --git "$repo" egent-code-plexus --bin ecp --locked --profile release-dist
```

## 🚀 Быстрый старт

Нет демона для запуска. Нет необходимой конфигурации. Одна команда — от нуля до запрашиваемого графа.

```bash
# Индексация (инкрементальная; первый запрос также автоматически индексирует при отсутствии индекса)
ecp admin index --repo .

# Найти символ — точное совпадение по умолчанию
ecp find loginUser
ecp find login --mode bm25            # BM25-ранжирование, разбитое на 5 выходных корзин

# Радиус взрыва — что сломается, если я это изменю?
ecp impact validateUser --direction upstream

# Полный контекст символа (сигнатура, тело, вызывающие, вызываемые, 1-шаговое воздействие)
ecp inspect validateUser

# Карта HTTP-маршрутов (декларативные @Get + императивные app.get())
ecp routes
ecp routes /api/users --method POST   # маршрут → обработчик → цепочка вызовов

# Использование файла: кто читает / записывает этот путь?
ecp impact --literal session_meta.json
```

Все команды на стороне чтения принимают `--format text|json|toon`. По умолчанию используется наиболее экономный по токенам формат для каждой команды (в основном `toon`; `find` по умолчанию использует `text`; `cypher`/`summary` по умолчанию используют `json`).

## 🛠️ Интерфейс командной строки

Два уровня: **агентские команды** на верхнем уровне (запрос / рефакторинг / верификация) и **административные команды** в `ecp admin` (реестр / хуки / деструктивные операции). Запустите `ecp --help` и `ecp admin --help` для полных матриц флагов.

**Агентские команды:**

| Команда | Назначение |
|---|---|
| `inspect <name>` | Символ → метаданные, декораторы, сигнатура, вызывающие, вызываемые, 1-шаговое воздействие, содержащиеся методы / свойства / варианты перечисления |
| `find <pattern>` | Точное · `--mode fuzzy` · `--mode bm25` (5 корзин: source / tests / reference / document / config) |
| `find-schema-bindings <field>` | Эвристические рёбра MirrorsField + кандидаты в blind-spot по классам / сервисам |
| `find-transaction-patterns [--class <Name>]` | Пары имён compensate/undo/rollback саги; ≥0.75 → POSSIBLY_RELATED, <0.75 → BLIND_SPOT |
| `impact <name> --direction <up\|down>` | BFS радиуса взрыва с фильтрацией по уверенности; `--since <ref>` для воздействия набора изменений |
| `rename --symbol <old> --new-name <new>` | Переименование с учётом AST в нескольких файлах для 14 языков. Всегда сначала `--dry-run`. |
| `cypher '<query>'` | Лазейка openCypher; `m.content` возвращает тело исходного кода |
| `summary` | Обзор реестра, покрытие фреймворков, каталог blind-spot для LLM, актуальность графа |
| `routes [<path>]` | Перечисление HTTP-маршрутов (декларативные + императивные); с `<path>`: обработчик + цепочка вызовов |
| `contracts` | Инвентаризация API-контрактов между репозиториями (маршруты / очереди / RPC) |
| `diff` | Дельта резолвера: деградация уровня привязки + изменения маршрутов / контрактов |
| `tool-map` | Места вызовов внешних HTTP / DB / Redis / очередей через анализ привязок импорта |
| `shape-check` | Дрейф между паттернами доступа HTTP-потребителей и формами ответов маршрутов |
| `peers` | Совместная работа нескольких сессий: `status / diff / say / inbox / log / thread / watch / gc` |
| `review` | Одноразовый аудит: impact + summary + tool-map + shape-check + diff, только высококонфидентные сигналы |

**Административные команды** (`ecp admin <cmd>`):

| Команда | Назначение |
|---|---|
| `index --repo <path>` | Построение / обновление графа; инкрементально через кэш содержимого xxh3_64. `--force` для полной перестройки. |
| `drop / prune / rename-branch` | Жизненный цикл индекса: удаление, обрезка устаревших директорий веток, переименование ветки на диске |
| `install-hook` | Хук транзакции git reference (автоматически отслеживает переключения веток) |
| `config` | Интерактивный мастер TOML для `.ecp/config.toml` |
| `mcp serve` / `mcp tools` | MCP-сервер (stdio); `tools` перечисляет открытый интерфейс |

Все команды разрешают `.ecp/graph.bin` из текущей директории, если не указан `--graph <path>`. Каждая команда, обращённая к агенту, неинтерактивна; каждый выходной поток поддаётся разбору.

### Синхронизация одноранговых узлов нескольких сессий

Когда несколько LLM-сессий редактируют один и тот же репозиторий параллельно, `ecp peers` отображает состояние грязных символов каждой сессии на уровне символов и обеспечивает прямой обмен сообщениями между сессиями. Регистрация через `ECP_SESSION_ID`, `CODEX_SESSION_ID`, `CODEX_THREAD_ID` или `CLAUDE_CODE_SESSION_ID`.

```bash
# Запустить наблюдатель (по одному на сессию; необходим для push-событий в inbox)
ecp peers watch --start

# Кто ещё редактирует прямо сейчас?
ecp peers status                                  # text
ecp peers status --format json                    # {session_id, pid, watcher: alive|dead|not-started}

# Просмотреть грязные символы узла
ecp peers diff <peer-session-id> [<symbol>]

# Отправить сообщения
ecp peers say "rebasing on main, hold pushes 5min"    # широковещательно
ecp peers say --to <peer-session-id> "take auth.rs?"  # адресно

# Чтение и управление
ecp peers inbox
ecp peers log --limit 20
ecp peers thread <msg-id>

# Очистка
ecp peers watch --stop && ecp peers gc
```

Поле `watcher` различает `alive` | `dead` | `not-started` — сбои не маскируются под «функция не используется».

### Доказуемые вердикты code review

`ecp review --verdicts` предвычисляет подкреплённые графом вердикты из секций `ecp diff`. Передайте JSON напрямую как контекст review — пропустите повторное вычисление LLM отношений вызывающих из сырого diff.

```bash
ecp review --since main --verdicts --format json
```

| Серьёзность | Правило |
|---|---|
| `RISK` | Существуют межфайловые вызывающие, публичный символ удалён или blindspot в регионе diff |
| `WARN` | Только внутрифайловые вызывающие или маршрут изменён |
| `INFO` | Вызывающие не найдены или добавлен новый публичный интерфейс |

Виды вердиктов: `SIGNATURE_OR_BODY_CHANGED` · `NEW_PUBLIC_SURFACE` · `REMOVED_PUBLIC_SURFACE` · `ROUTE_CONTRACT_CHANGED` · `BLINDSPOT_IN_DIFF_REGION`

Каждый вердикт ссылается на точную секцию diff и факт графа, который его вызвал. Полная спецификация: [docs/specs/2026-05-22-review-verdicts.md](../specs/2026-05-22-review-verdicts.md).

## 🔌 Интеграция с агентами

**Предпочтительнее нативный путь**, где он доступен — он подключает хуки авто-переиндексации и навыки рабочего процесса, которые учат агента *когда* запросы к графу стоят затраченного времени. **MCP — универсальный запасной вариант** для любого хоста, поддерживающего протокол.

| Агент | Путь | Подключает |
|---|---|---|
| Claude Code | нативный | хуки + навыки + опциональный MCP |
| Codex CLI | нативный | навыки (нативные инструменты в разработке) |
| Gemini CLI | нативный | нативный навык **или** MCP |
| Cursor · Windsurf · Cline · Copilot · любой MCP-хост | MCP | MCP-сервер |

Пошаговая настройка: `ecp admin → Agent Integrations → <host>`. Скриптовый путь для автоматизации: `ecp admin <host> install <component>`. Проверка любого хоста: `ecp admin <host> status`.

### Claude Code

```bash
ecp admin claude install hooks          # settings.json: авто-переиндексация + обогащение контекста
ecp admin claude install skills all     # пакеты навыков ecp + simplify (или: ecp | simplify)
ecp admin claude install mcp-server     # опционально — хуки + навыки + CLI уже достаточно
```

Хуки подают контекст графа при каждом Grep/Glob/Bash без явного вызова инструмента. Навык `ecp` обучает рабочим процессам с символами / воздействием / маршрутами / контрактами / переименованием. `simplify` управляет code review на основе графа.

### Gemini CLI

```bash
ecp admin gemini install native-skill   # подключение через `gemini skills link`
ecp admin gemini install mcp-server     # регистрация через `gemini mcp add`
```

`native-skill` и `mcp-server` взаимно исключают друг друга — установка одного удаляет другой.

### Codex CLI

```bash
ecp admin codex install skills all      # ecp + simplify; нативные инструменты ждут подключения Codex
```

**Навыки рабочих процессов:**

| Навык | Когда использовать |
|---|---|
| `ecp` | Агент решает, превосходят ли рабочие процессы на основе графа grep / чтение файлов для символов, вызывающих, маршрутов, контрактов |
| `simplify` | Code review начиная с воздействия ecp, слепых пятен, исходящих вызовов, дрейфа форм, дельт резолвера |

### MCP запасной вариант (Cursor, Windsurf, Cline, любой MCP-хост)

| Хост | Файл конфигурации |
|---|---|

## 🏗️ Архитектура

```
crates/
├── ecp-core        # Zero-copy граф (rkyv + mmap), инкрементальный кэш, запросы к графу
├── ecp-analyzer    # Парсеры tree-sitter, детектор HTTP-маршрутов, уверенность фреймворков
├── ecp-mcp         # MCP-сервер (stdio) — открывает основные команды как инструменты
└── ecp-cli         # Бинарный файл `ecp`, движок Tantivy BM25, оптимизированный вывод
```

Разбор → разрешение → сериализация проходит через канал MPSC в один поток-строитель, который собирает граф и записывает zero-copy `.ecp/graph.bin`. Пути чтения (`inspect`, `cypher`, `impact`, …) выполняют mmap этого файла напрямую — без шага десериализации. Кэш содержимого xxh3_64 обеспечивает инкрементальные перестройки менее чем за секунду на репозитории из 22 тысяч файлов.
